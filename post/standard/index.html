代码风格规范化是一个程序员必备的素质

代码规范的意义：

以最低程度的限制，来统一各个项目的编码规范，避免各家自行发展的风格阻碍了程序设计师开发的困扰。PSR 是 Proposing a Standards Recommendation (提出标准建议) 的缩写。也正是基于此，才会出现后面composer，使php的生态圈开始形成。

PSR 原来有五个规范，分别是：

  • PSR-0 (Autoloading Standard) 自动加载标准。2014-10-21该标准已经被废弃，使用PSR-4替代，不再细讲
  • PSR-1 (Basic Coding Standard) 基础编码标准。
  • PSR-2 (Coding Style Guide) 编码风格向导。
  • PSR-3 (Logger Interface) 日志接口。
  • PSR-4 (Improved Autoloading) 自动加载的增强版，可以替换掉 PSR-0 了。

今天我们的代码规范是基于以上规范进行了整理。

一 代码格式规范

1、PHP 源文件只能使用 <?php 和 <?= 这两种标签

<?php 标签通常用于纯 PHP 的脚本当中，而 <?= 通常用于模板当中。

2、PHP 源文件必须是不带 BOM 的 UTF-8 编码的文件

BOM（Byte Order Mark)，字节顺序标记，出现在文本文件头部，Unicode 编码标准中用于标识文件是采用哪种格式的编码。

3、PHP 源文件缩进采用 4 个空格

很多编辑器使用 Tab 作为缩进。会造成空格性问题。

4、纯 PHP 代码的源文件关闭标签 ?> 必须省略

PHP 解析器在对文件进行解释的时候，会有性能提升。并且，这能一定程序避免在 ?> 之后有多余的空格导致程序报错。

5、请严格控制每行 120 个字符

过长的代码会导致多种分辨率的显示器造成兼容问题。并且，过长的代码也会造成难以阅读理解。如果实在太长，请把代码换行。

6、所有的类必须设定一个命令空间，且命名空间(namespace)的声明后面必须有一行空行

命令空间给代码结构有较强的说明性，以及杜绝同名类的冲突问题。同时，也能用到 Composer 的自动加载优势特性。

7、所有的导入(use)声明必须放在命名空间(namespace)声明的下面，一句声明中，必须只有一个导入(use)关键字，且use代码块后必须空一行

12、控制结构花括号、换行、空格等规范

if、else、elseif、switch、for、foreach、case、while、go、try、catch 等关键词后面必须加空格。可以说，没有特殊说明的情况下，基本上所有的 PHP 关键词后面都必须加空格。

二 代码命名规范

1、PHP 关键字必须小写

boolean 值：true，false，null 也必须小写。下面的关键字，也必须小写： ‘__halt_compiler’, ‘abstract’, ‘and’, ‘array’, ‘as’, ‘break’, ‘callable’, ‘case’, ‘catch’, ‘class’, ‘clone’, ‘const’, ‘continue’, ‘declare’, ‘default’, ‘die’, ‘do’, ‘echo’, ‘else’, ‘elseif’, ‘empty’, ‘enddeclare’, ‘endfor’, ‘endforeach’, ‘endif’, ‘endswitch’, ‘endwhile’, ‘eval’, ‘exit’, ‘extends’, ‘final’, ‘for’, ‘foreach’, ‘function’, ‘global’, ‘goto’, ‘if’, ‘implements’, ‘include’, ‘include_once’, ‘instanceof’, ‘insteadof’, ‘interface’, ‘isset’, ‘list’, ‘namespace’, ‘new’, ‘or’, ‘print’, ‘private’, ‘protected’, ‘public’, ‘require’, ‘require_once’, ‘return’, ‘static’, ‘switch’, ‘throw’, ‘trait’, ‘try’, ‘unset’, ‘use’, ‘var’, ‘while’, ‘xor’

2、类名必须与文件名一样

3、类的命名必须遵循 StudlyCaps 大写开头的驼峰命名规范

4、方法名称必须符合 camelCase 式的小写开头驼峰命名规范

5、类中的常量所有字母都必须大写，单词间用下划线分隔

6、变量必须使用小驼峰命名风格

7、杜绝无意义的参数名和数值

9、方法的代码行数尽量控制在40行之内

如果发现自己的代码超过了 40 行，那么就需要考虑自己的代码是不是有拆分不合理的地方。特殊情况允许超过 40 行。但是，整个方法里面的代码必须是简单的判断逻辑。不包含复杂的业务判断逻辑。因为，不同的业务判断最佳实践是单独封装一个方法。

三 代码层级规范

采用 msvc 开发模式，与传统的 mvc 模式多了一个 service层

Model：

  • 关联关系(relation)
  • 表属性(property)
  • 模型事件(model event)
  • 访问器、修改器
  • 序列化

Service：

  • 业务逻辑处理
  • 事务
  • 缓存
  • 抛出异常

View：

  • 数据渲染

Controller：

  • 参数数据验证（大型项目中在请求中间件中处理或网关中处理）

  • 对 Service 的各种情况进行处理，理想情况下，Service 中出现的所有非理想结果都应该抛出异常，由 Controller 进行捕获后处理给用户。

  • 返回想要的数据格式（json或html）

四 代码提交流程规范

1、commit message 规范

  • feat : 新功能
  • fix：修复bug
  • refactory：方法重构
  • docs:修改文档

eg： git commit -m 'feat:新功能'

2、没有完善及后期需要优化的代码打上 todo标签，后期利用编辑器可以统一查看，以免遗忘

3、代码提交到 自己的远程 分支后，需要在 gitlab 上提交 merge request ，由代码审核人通过后，方能合并到 develop

4、代码提交时加入了代码检测工具，主要检测代码格式是否符合国际规范(PSR-1,PSR-2)

在项目根目录下执行 ./vendor/bin/phpcbf –standard=PSR2,PSR1 frontend/controllers/ArticleController.php，即可修复代码格式。注意超出 120 字符的错误需要自己手动修复

5、代码拉取采用 git pull --rebase origin develop

git rebase 的好处可以自行百度
